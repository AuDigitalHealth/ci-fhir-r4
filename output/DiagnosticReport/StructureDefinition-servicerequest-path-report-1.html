SD.5 StructureDefinition-servicerequest-path-report-1 [Draft]

Order Details for Pathology Report

The purpose of this profile is to represent key elements of clinical relevance of a request for a pathology diagnostic investigation in a pathology report context. This profile is intended to support the electronic exchange of pathology reports between healthcare providers, and between healthcare providers and the My Health Record system infrastructure in Australia.

The context of this profile is reporting.

Implementation guidance

For the overarching usage scenarios in this implementation guide it is expected that:

  • if sent, one identifier is the local identifier assigned to the order by the order requester (i.e. placer identifier)
  • a local identifier is sent with a HPI-O scoped identifier namespace if there isn’t a local namespace available (see the FAQ for more information)
  • status is ‘completed’

When sending to the My Health Record system it is expected that:

  • patient conforms to My Health Record Patient
  • requester is sent as a reference to a PractitionerRole resource with PractitionerRole.practitioner as either:
    • a reference to a Practitioner resource with Practitioner.name.family, or
    • practitioner.display with the at least the practitioner's family name

Boundaries and relationships

Requesting of pathology services is not supported by this implementation guide and may in future be handled in a referral or service requesting implementation guide.

This profile is referenced by My Health Record Pathology Report, and Atomic Pathology Report.

SD.5.2 - Known Issues

Reference Description Issue No.
status (terminology binding)

Publication of the RequestStatus Reporting value set is pending.

 See ci-fhir-r4/issues/60
category (pattern)

The use of pattern on category is in conflict with the intended design and implementation guidance. We want to fix a domain category, e.g. imaging, and encourage the sending of a second finer grained category, but current profile forces all instances of category to match the fixed pattern or an error is thrown. We need to change this design, possibly slicing on category or using an invariant would be better.

 See ci-fhir-r4/issues/87
category (potential code system redundancy / duplication)

The element category is present in DiagnosticReport, Observation, and ServiceRequest. A rationalisation of the codes used for categorisation needs to be performed.

Observation.category: "laboratory" http://terminology.hl7.org/CodeSystem/observation-category

DiagnosticReport.category: "LAB" http://terminology.hl7.org/CodeSystem/v2-0074

ServiceRequest.category: "108252007" https://snomed.info/sct (Laboratory procedure)

 See ci-fhir-r4/issues/28
code (RCPA terminology)

One or more value sets representing the Standard for Pathology Informatics in Australia - Requesting codes (see DOWNLOADS) are in progress. This terminology is expected to be included in the HL7 AU Base Implementation Guide in AU Diagnostic Service Request.

When this content is available we are expecting to replace the placeholder SNOMED CT-AU set with this material.

 See au-fhir-base/issues/401
Support for screening programs initiated requests Not yet clear how requests that are initiated by a national program, e.g. National Bowel Cancer Screening Program, are well supported beyond allowing an Organization resource to be sent in requester. We need to investigation how this can be supported and include that information in this guide, e.g. possibly with slicing or examples or guidance or Organization profiles. See ci-fhir-r4/issues/29
Support for self initiated requests Not yet clear how requests that are initiated by a patient or related person should be supported. ServiceRequest.requester does allow for a Patient or RelatedPerson resource though currently constrained out in this profile. We need to investigate the need for supporting self-initiated requests in the usage scenarios this profile is intended to support. See ci-fhir-r4/issues/56
Constraint presentation The full set of constraints (i.e. invariants) defined in this profile are only presented in the Detailed Descriptions tab or the raw representation (e.g. XML) of the profile. The Differential Table only presents constraints introduced in this profile in addition to the constraints present in the base profile and base resource. The Snapshot Table only presents the constraints visible in the Differential Table and additionally presents those constraints set in slices in the base profile.
