SD.8.1 Formal Views of Profile Content

The official URL for this profile is: 

http://ns.electronichealth.net.au/ci/fhir/StructureDefinition/diagnosticreport-otherdiag-atomic-1

The purpose of this profile is to define an atomic representation of a report as issued by the diagnostic service provider to support the electronic exchange of other diagnostic investigations (such as echo cardiogram or colonoscopy or hearing test) between healthcare providers in Australia.

Must Support

In the context of this profile Must Support SHALL be interpreted as follows:

  • The system SHALL be able to include the element when authoring a resource where data is available

  • The system SHALL be able to store and retrieve the element

  • The system SHALL be able to process resources containing the element

This profile builds on AUBaseDiagnosticReport.

This profile was published on Fri May 22 11:01:42 AEST 2020 as a draft by Australian Digital Health Agency.

SD.8.2 - Known Issues

Reference Description Issue No.
Usage of "Atomic" in description The term "Atomic" is used to describe data at the element level allowing element names and values to be represented in the report. Whether "Atomic" is the appropriate descriptive term to be used needs to be determined by the stakeholders and changes made if necessary. See ci-fhir-r4/issues/27
meta.profile (invariant on profile value) Invariant enforcing the canonical url for this profile is missing - work in progress. See ci-fhir-r4/issues/53
status (terminology binding)

We need to create a more constrained value set for reporting the status of a diagnostic report for the overarching usage scenarios - this value set would be used for pathology, imaging, and other diagnostics.

Thinking to constrain the status with the current required valueset of http://hl7.org/fhir/R4/valueset-diagnostic-report-status.html to just contain the codes of "partial", "preliminary", "final", "amended", "corrected", "appended" i.e remove "registered", "cancelled", "entered-in-error" & "unknown".

 See ci-fhir-r4/issues/53, and jira.aws.tooling/browse/FTR-933
category (terminology binding) We need to create a more constrained value set that at least removes imaging and pathology. See ci-fhir-r4/issues/42
category (fixed code for implementation) There is likely a need for a single fixed code to indicate that this is 'other diagnostics' for a similar use case in the My Health Record to identifying and retrieving 'pathology' and 'imaging'. See ci-fhir-r4/issues/42
code (missing terminology)

We need a value set whose permissible values represent types of diagnostic tests that are not categorised as either a pathology test nor a diagnostic imaging test. Common examples of these tests are:

  • cardiology field - ECG, echo cardiogram, angiocardiography
  • neurology field - EEG, EMG, evoked potentials, nerve conduction tests
  • gastroenterology field - endoscopy, colonoscopy
  • respiratory field - pulmonary function tests
  • audiology - hearing tests
  • sleep studies

These types of tests are those that are not done by either pathology or imaging providers, and are usually done by specialist providers (ie cardiologists, neurologists etc). And apart from the common tests above, it is expected that the relevant providers would have many more specialised tests that they would perform.

This terminology is expected to be included in the HL7 AU Base Implementation Guide in AU Base Diagnostic Report

 See jira.aws.tooling/browse/FTR-898, and au-fhir-base/issues/402
subject (Reference type too open) The Reference type is too open, we need to ensure that either a reference conforming to the profiles or a logical reference via identifier is supplied - work in progress. See ci-fhir-r4/issues/53
performer (Reference type too open) The Reference type is too open, we need to ensure that either a reference conforming to the profiles or a logical reference via identifier is supplied - work in progress. See ci-fhir-r4/issues/53
